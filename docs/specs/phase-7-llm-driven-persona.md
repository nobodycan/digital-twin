# Phase 7 LLM-Driven Persona Spec

Date: 2026-06-23

Status: Approved by the user on 2026-06-23. Stage 3 implementation complete, Stage 4 review complete, Stage 5 local QA complete on 2026-06-24, and Stage 6 security check found no high-severity blockers. Pending Stage 7 ship/integration.

## Context

Phase 0-6A completed a local-first professional digital-human foundation: typed contracts, local storage, persona/router/skills/agents, orchestrator, CLI/HTTP/SSE, Web/Admin surfaces, governance/evals, readiness, provider-shaped deployment, and fake-server-tested provider boundaries.

The most visible product gap is now clear from local usage:

- User asks: "你是谁"
- Assistant always replies: "I'm here and keeping the same professional persona."

That reply is not generated by a model. It is a deterministic placeholder in `PersonaAgent.Run`. Phase 7 should make persona chat genuinely LLM-driven while preserving the local/mock safety and deterministic tests that made earlier phases stable.

## Office-Hours Framing

The tempting framing is: "Connect a real model everywhere." That is too broad.

The better Phase 7 question is:

> What is the smallest LLM integration that makes the digital human feel genuinely responsive while keeping provider failures, secrets, tests, and governance under control?

## Premise Challenge

- LLM-driven persona chat is the highest-value first cut because it fixes the exact user-visible failure: every general question receives the same placeholder.
- Router, RAG, memory, tool planning, and voice can remain deterministic in the first slice. If they all become LLM-driven at once, debugging becomes much harder.
- The product must stay honest about model identity. If no model is configured, the assistant should say it is running in local deterministic mode. If an LLM is configured, metadata can name the configured provider/model without exposing secrets.
- Tests must not call real OpenAI, Claude, or any paid provider. Fake server tests and fake `llm.Client` implementations remain mandatory.
- Local/mock must remain the default so a fresh clone still works without credentials.
- Governance must run after generation, not only before generation. Persona guard and safety checks should be able to block or replace unsafe model output.

## Recommended Scope: Phase 7A Real Persona Generation MVP

Phase 7A should focus on LLM-backed persona responses for `persona.chat`.

### In Scope

- Add configurable LLM provider fields for chat generation: provider, base URL, model, API key, timeout, and enabled/fallback policy.
- Add an LLM client factory that returns:
  - deterministic local/mock client by default;
  - OpenAI-compatible HTTP client when explicitly configured.
- Extend `PersonaAgent` so it can call an `llm.Client` for persona chat generation.
- Build the prompt from the existing persona renderer plus conversation context.
- Preserve deterministic fallback when no LLM is configured or provider fails.
- Apply persona guard to generated output and return safe fallback when guard rejects output.
- Expose provider/model metadata in `AgentResult.Metadata` without leaking secrets.
- Add tests with fake `llm.Client` and `httptest.Server`; no real network in CI.
- Update README and release notes to explain local/mock mode vs configured LLM mode.

### Out of Scope

- Full LLM planning across every Agent.
- Real tool-use reasoning or autonomous tool selection.
- RAG answer generation over uploaded knowledge.
- Long-term memory summarization changes beyond existing `llm.Client` contracts.
- Real TTS/ASR/avatar provider work.
- Streaming token-by-token UI as a hard requirement. Existing SSE can continue to emit completed messages; token streaming can be Phase 7B.
- SQLite/Postgres, queues, Kubernetes, OAuth/RBAC, billing, or compliance certification.
- Any automated test that requires paid provider credentials.

## User Value

After Phase 7A, a user should be able to ask general questions like:

- "你是谁"
- "你背后是什么模型"
- "帮我分析一下今天的工作安排"
- "用更专业但不生硬的语气回复我"

and receive context-aware answers instead of a fixed placeholder.

The system should also stay transparent:

- In local/mock mode, it should not pretend to be backed by a real model.
- In configured LLM mode, it should report configured provider/model metadata safely.
- On provider failure, it should degrade gracefully and record why.

## Hidden Assumptions

- Existing `internal/llm.OpenAIClient` is good enough as a first OpenAI-compatible adapter, but Stage 2 should verify whether it needs timeout/config hardening before use in `cmd/server`.
- Existing `persona.Renderer` can be reused for system prompt construction.
- Existing `persona.Guard` is enough for first-pass post-generation enforcement; richer safety evaluation can follow in Phase 7B.
- Existing `/chat`, `/chat/stream`, and `/experience/stream` should not need route changes for Phase 7A.
- Config should keep local/mock as default to avoid breaking existing smoke tests and developer onboarding.

## Alternatives Considered

### Approach A: LLM-Driven PersonaAgent Only (Recommended)

Wire LLM generation only into persona chat. Keep router, tools, knowledge, memory, and voice deterministic.

Completeness: 8/10 for the current user-visible problem.

This directly fixes the fixed-response issue while keeping the blast radius small and testable.

### Approach B: LLM Router Plus Persona Generation

Enable LLM classification and persona generation together.

Completeness: 7/10.

This makes routing more flexible, but it introduces two LLM failure modes at once: wrong intent and bad answer. It is better after persona generation is stable.

### Approach C: Full LLM Agent Orchestration

Let the LLM decide agent/tool flow, generate answers, and choose skills.

Completeness: 10/10 for ambition, 4/10 for Phase 7 safety.

This is closer to a true agentic digital human, but it risks breaking deterministic governance and tool authorization boundaries too early.

### Approach D: Real Voice/Avatar Provider First

Make the digital human sound or look better before improving answer quality.

Completeness: 5/10.

This improves presentation, but the core issue remains: the assistant would still say the same placeholder in a nicer voice.

## Acceptance Criteria for the Future Phase 7 Plan

Stage 2 must produce a plan that:

- Keeps local/mock mode as the default.
- Defines LLM config fields and validation without leaking secrets.
- States exactly which provider adapter is supported first.
- Adds fake-server or fake-client tests for successful generation, provider failure, malformed response, timeout, guard rejection, and fallback.
- Proves `PersonaAgent` no longer returns the hard-coded placeholder when an LLM client is configured.
- Proves existing CLI/HTTP/SSE paths remain compatible.
- Documents how to configure LLM mode locally.
- Documents that CI does not call real providers.

## Open Questions for Stage 2

- Should Phase 7A use the existing `llm.OpenAIClient` directly, or add a small factory/wrapper first?
- Should LLM config reuse `LLMConfig` or use the generic `ProviderConfig` shape already used by TTS/ASR?
- Should provider failure fallback always return deterministic mock output, or fail closed in production-like profiles?
- Should the assistant answer "what model are you" from configured metadata, or should that response be a hard-coded transparency rule before LLM generation?
- Should Phase 7A include non-streaming generation only, or also wire `llm.Stream` into `/chat/stream`?
