# Phase 9 Experience and Provider Diagnostics Spec

Date: 2026-06-27

Status: Approved by the user on 2026-06-27

Mode: SDD Stage 1 / gstack office-hours

## Context

Phase 7 and Phase 8 moved the system from deterministic placeholder replies to a
real provider-shaped, durable streaming conversation loop. The current product
surface still feels like an engineering test harness:

- the Web app says "Digital Human Console" but visually reads as a plain demo page;
- the avatar area only shows state text such as `idle`, `thinking`, or `speaking`;
- the transcript does not explain which model/provider is active;
- provider failures collapse into a repeated assistant message:
  `I hit a provider issue, so I'm falling back to a local safe reply for now.`;
- local fallback is safe, but not actionable for the user who is trying to verify
  DeepSeek-backed behavior.

The next phase should make the project usable as a local professional digital-human
experience: visually credible, operationally honest, and easy to debug when a real
LLM provider is misconfigured.

## Goal

Deliver a polished first-use Web experience and a provider-diagnostics path that:

1. makes the digital-human page look intentional and professional;
2. shows conversation progress, provider mode, model name, and recoverable errors;
3. prevents repeated generic fallback replies from looking like normal assistant
   answers;
4. gives developers enough local diagnostics to fix DeepSeek/OpenAI-compatible
   configuration without leaking secrets;
5. remains local-first and deterministic in CI.

## Premise Challenge

The request has two visible symptoms: "the frontend is ugly" and "the assistant
keeps saying provider fallback." The tempting implementation is to restyle the page
and change the fallback sentence. That is too shallow.

The deeper product problem is trust. A professional digital human must make three
states clearly different:

- **live answer**: generated by the configured provider/model;
- **local fallback**: safe, deterministic recovery because the provider failed
  before any visible output;
- **hard error**: no trustworthy assistant answer was produced.

If these states all appear as ordinary assistant messages, the user cannot tell
whether the system is working. A prettier page would only make the failure more
confusing.

## Recommended Approach

Ship Phase 9 as one user-facing reliability slice:

- redesign `/app` into a focused two-pane digital-human workspace;
- add lightweight provider/session status to the UI;
- add sanitized provider diagnostics to stream events and server responses;
- change fallback rendering so local fallback is visually labeled and actionable;
- tighten OpenAI-compatible stream completion semantics for DeepSeek-style behavior;
- add opt-in smoke tooling that can prove whether the configured provider works
  before the user opens the browser.

This keeps scope narrow while solving the user's actual confusion.

## In Scope

### Web Experience Polish

- Replace the current bare console styling with a refined local product UI:
  - conversation-first layout;
  - restrained professional palette with clear contrast;
  - stable responsive layout for desktop and mobile;
  - a more expressive avatar state surface without adding real 3D/Live2D providers;
  - visible state chips for `ready`, `thinking`, `speaking`, `fallback`, and `error`;
  - a compact session/provider strip.
- Keep the app as a tool, not a marketing landing page.
- Keep `/admin` functional; visual polish can focus on `/app` first.

### Provider Status and Diagnostics

- Expose sanitized runtime/provider metadata in UI-friendly form:
  - provider name;
  - model name;
  - fallback policy;
  - generation mode: `llm`, `local`, `fallback`, `transparency`;
  - safe error category such as `provider_status`, `provider_stream_decode`,
    `provider_stream_truncated`, `provider_empty_response`, or `provider_network`.
- Do not expose API keys, raw provider error bodies, Authorization headers, or
  unredacted URLs with credentials.
- Add a small diagnostics endpoint or extend existing app bootstrap data so `/app`
  can show "DeepSeek configured" versus "local deterministic mode" before the first
  message.
- Surface provider failure as status UI plus a transcript system line, not as an
  unlabeled normal assistant answer.

### Fallback Semantics

- Keep `fallback_to_local` as a valid local-first policy.
- Make fallback explicit:
  - the assistant response may still be shown when safe and complete;
  - the UI must label it as local fallback;
  - the status message must explain that the configured provider did not produce
    a usable response.
- Preserve `fail_closed` for users who prefer a hard error over a fallback answer.
- Improve fallback copy so it is helpful in Chinese and English contexts.

### DeepSeek/OpenAI-Compatible Streaming

- Verify request shape against OpenAI-compatible expectations:
  - `POST {base_url}/chat/completions`;
  - `Authorization: Bearer <api-key>`;
  - configured model;
  - `stream: true` for streaming paths.
- Treat malformed chunks and empty streams as provider failures.
- Revisit EOF-without-explicit-done behavior. If the provider emits partial content
  and then closes without a terminal marker, the system should classify it as a
  truncated provider stream unless an explicitly documented compatibility mode says
  otherwise.
- Add tests around DeepSeek-like streaming fixtures using fake HTTP servers only.

### Developer UX

- Improve `scripts/start-deepseek.ps1` output so it reports:
  - selected port;
  - provider/model/base URL;
  - fallback policy;
  - smoke command to run;
  - browser URL.
- Add or improve a provider smoke script that performs one streaming turn and exits
  non-zero with a sanitized diagnostic if provider setup is wrong.
- Document common local failure causes:
  - port already in use;
  - missing or expired API key;
  - wrong base URL;
  - wrong model name;
  - fallback policy hiding real provider failures.

## Out of Scope

- Real 3D avatar, Live2D, video avatar, real TTS, or real ASR provider integration.
- OAuth, user login, production RBAC, billing, quotas, or public deployment.
- Provider-specific SDKs. Continue using the OpenAI-compatible HTTP boundary.
- RAG answer generation, autonomous tool planning, or new Agent capabilities.
- Replacing local file storage with SQLite or another database.
- Running real DeepSeek calls in CI.

## Alternatives Considered

### Alternative A: Visual Refresh Only

This would restyle `/app` and leave provider fallback behavior mostly unchanged.

Pros:

- fastest path to a nicer screenshot;
- low backend risk;
- easy to keep tests focused on static assets.

Cons:

- does not explain why DeepSeek is not producing answers;
- repeated fallback still looks like product behavior;
- users may trust a polished but non-working assistant.

Verdict: insufficient.

### Alternative B: Provider Fix Only

This would focus on DeepSeek compatibility, stream completion, and fallback policy,
with minimal UI changes.

Pros:

- attacks the most urgent functional issue;
- improves backend correctness;
- easier to validate through Go tests.

Cons:

- the product would still feel like a debugging panel;
- users still lack clear visual state and recovery cues;
- "professional digital human" remains mostly theoretical.

Verdict: useful but too narrow.

### Alternative C: Experience and Diagnostics Slice

This combines front-end polish with provider observability and clearer fallback
semantics.

Pros:

- directly addresses both user-visible complaints;
- turns errors into actionable local debugging feedback;
- strengthens trust without expanding into heavy avatar/provider work.

Cons:

- touches both Web and backend code;
- requires careful tests so visual work does not mask provider behavior;
- needs a clear boundary to avoid becoming a full redesign.

Verdict: recommended.

## Acceptance Criteria

- `/app` presents a polished, responsive professional digital-human workspace.
- A configured provider/model is visible in the UI without exposing secrets.
- Local deterministic mode is clearly distinguishable from real LLM mode.
- Provider failure before visible output does not appear as an unlabeled ordinary
  assistant answer.
- A DeepSeek-like fake server can prove streaming success, malformed stream failure,
  empty stream failure, and truncated stream classification.
- `fallback_to_local` and `fail_closed` are both covered by tests.
- `scripts/start-deepseek.ps1` and smoke tooling give the user concrete next commands.
- No real API keys or provider calls are required in CI.

## Test Matrix Seed

| Area | Scenario | Expected Result |
| --- | --- | --- |
| Web UI | Desktop `/app` load | No overlap, professional layout, provider strip visible |
| Web UI | Mobile `/app` load | Composer, transcript, and avatar/status areas remain usable |
| Web UI | LLM success stream | Deltas append to one assistant turn and final status is `done` |
| Web UI | Pre-output fallback | Assistant/fallback line is labeled and status explains provider failure |
| Web UI | Stream error after visible output | Partial output is marked not saved and no fallback text is appended |
| Provider | Fake DeepSeek success | Request shape includes configured model and streaming succeeds |
| Provider | Non-2xx response | Sanitized error category, no secret leakage |
| Provider | Malformed SSE chunk | Provider failure, no persisted assistant answer |
| Provider | Empty stream | Provider failure or labeled local fallback depending on policy |
| Provider | EOF without done | Classified consistently by documented compatibility rule |
| Scripts | start script | Prints selected port, provider/model, fallback policy, app URL, smoke command |
| Scripts | smoke script with bad URL | Exits non-zero with sanitized diagnostic |

## Open Questions for Stage 2

1. Should Phase 9 default DeepSeek startup to `fail_closed` during local verification
   so provider errors are impossible to miss, while keeping app config default as
   `fallback_to_local`?
2. Should EOF-without-`[DONE]` be universally treated as truncated, or should we add
   an explicit compatibility flag for providers that close cleanly without `[DONE]`?
3. Should `/app` consume a new `/runtime/status` endpoint, or should status be
   embedded in the first stream event to avoid adding another HTTP surface?

## Recommended Assignment Before Stage 2

Run one local DeepSeek check with fallback disabled so the next plan can be grounded
in the real provider error:

```powershell
$env:DIGITAL_TWIN_LLM_FALLBACK_POLICY="fail_closed"
.\scripts\start-deepseek.ps1
.\scripts\smoke-conversation.ps1 -BaseUrl http://localhost:18080
```

If it fails, keep the sanitized output. Stage 2 should use that concrete failure
when locking the implementation plan.
